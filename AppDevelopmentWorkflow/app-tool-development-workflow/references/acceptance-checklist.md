# Acceptance Checklist

## 1. 需求與規格

- [ ] `project-brief.md` 已描述問題、使用者、成功標準與範圍。
- [ ] `system-spec.md` 已列出功能需求與驗收條件。
- [ ] 重要功能都有可測試的 acceptance criteria。
- [ ] Out of scope 已明確記錄。
- [ ] 待確認事項已列出，且不影響本次交付的項目已標註。

## 2. 技術設計

- [ ] `technical-design.md` 已描述技術棧與架構。
- [ ] 主要模組責任清楚。
- [ ] 資料來源、資料模型與保存方式清楚。
- [ ] API / CLI / UI 輸入輸出清楚。
- [ ] 設定值與 secrets 沒有硬寫在程式碼中。

## 3. 實作

- [ ] 最小可用流程可完整跑通。
- [ ] 正常流程可用。
- [ ] 空資料情境可處理。
- [ ] 錯誤輸入可處理。
- [ ] 外部服務失敗可處理。
- [ ] 權限或設定缺失可處理。
- [ ] 重複執行不會造成不可接受的副作用。

## 4. 測試

- [ ] 自動化測試已執行，或已記錄未能執行的原因。
- [ ] 手動驗收已完成。
- [ ] UI 已檢查主要桌面與手機尺寸，若本專案有 UI。
- [ ] CLI / API 已驗證主要參數、錯誤訊息與輸出格式。
- [ ] 重要 bug 已修復或列入已知問題。

## 5. 使用文件

- [ ] `README.md` 已說明安裝方式。
- [ ] `README.md` 已說明設定方式。
- [ ] `README.md` 已說明啟動與使用方式。
- [ ] `README.md` 已說明測試方式。
- [ ] `README.md` 已列出常見錯誤與處理方式。

## 6. 維護與迭代

- [ ] Log、資料保存與備份位置清楚。
- [ ] 有更新流程或版本紀錄。
- [ ] 後續需求已列入 roadmap 或 issue。
- [ ] 已知限制已記錄。
- [ ] 未來維護者可以依文件重現主要使用流程。
